01 — Overview

What the model does

The application allocates overhead from raw General Ledger accounts through to service lines using a five-stage sequential apportionment engine. It transforms unstructured cost data into a fully reconciled service-line cost view — the terminal question being "what does each service line actually cost, once every shared overhead has been fairly attributed?"

The apportionment module is self-contained. It lives inside a broader Pigment application that also covers demand planning and supply chain, among other domains, but sits in its own folder prefixed 9. CST Apportionment, with every block prefixed CST_ for easy identification. This write-up covers the cost module only.

Built in Pigment — a cloud-native business planning platform — the model leans on a handful of platform primitives. The next section defines them, so the rest reads cleanly whether or not you build in Pigment day to day.

03 — Structure

The dimension model

Dimensions are the axes that structure every piece of cost data. All twelve live in /9.1 Dimensions.

DimensionItemsPurpose
CST_Account36GL accounts — the source of all cost data. Properties: Account_Description, Account_Type.
CST_CostCenter18Organisational cost centres. Carries a Cost Pool property linking each centre to its source pool, and a calculated Cost Pool (Dest) that mirrors it to the destination axis.
CST_CostPool10Intermediate buckets that aggregate account cost before distributing to activities.
CST_CostPoolDest10A mirror of CST_CostPool used as the destination axis in Pool-to-Pool settlement. This dual-dimension pattern avoids circular references.
CST_Activity12Business activities — the layer between pools and service lines.
CST_ActivityDest12Mirror of CST_Activity used as the destination axis in Activity-to-Activity settlement.
CST_ServiceLine9Final cost destinations — the service lines that receive allocated overhead.
CST_Driver11Allocation drivers (headcount, FTE, floor space). Define how cost is split between pools, activities, or service lines.
CST_ApportionmentType3Classifies each account as Direct, Indirect, or other. Drives the branching at Stage 1.
CST_ReconciliationCheck13Named checks (RC01–RC06, VAL00–VAL05). The axis for all control reporting. Properties: Check_Type, Check_Description.
CST_Version3Planning versions — Actual, Budget, Forecast. Properties: Start_Period, End_Period, Version_Type.
CST_Stage2Flags the two sides of reciprocal settlement (Source / Destination).

Two design patterns worth stealing

Pattern · Dual-dimension for reciprocal settlement

CST_CostPool / CST_CostPoolDest and CST_Activity / CST_ActivityDest are identical in membership but represent the source and destination roles. This is the standard Pigment way to model many-to-many cost flows without creating circular formula dependencies.

Pattern · Sentinel items

Several dimensions carry a special "Input" item (for example in CST_ActivityDest and CST_CostCenter). This is a deliberate modelling sentinel: it stores a pre-split total at a known, addressable coordinate in the metric space, which the downstream formula then selects and redistributes via Pigment's SELECT modifier — avoiding a set of separate staging metrics.

05 — The engine

The five-stage apportionment chain

Cost moves through five sequential stages. Each stage is a distinct Pigment calculation that feeds the next; two of them (1b and 2b) handle reciprocal settlement and are computed externally, because Pigment does not support native circular or iterative formulas.

1

Account → Pool

Distribute raw account balances to cost pools
Board Stage 1 — Account to Pool  ·  Folder 9.2 Config + 9.3 Calculation

Raw GL balances are distributed to cost pools using driver percentage shares. Accounts classified Indirect are spread proportionally across pools. Accounts classified Direct bypass the chain entirely and flow straight to service lines at the end.

MetricTypeDimensionsRole
CST_Input_Account_Amount ($)Manual inputAccount, CostCenter, Version, MonthRaw GL amounts from CST_TXN_GL_Account
CST_Input_Apportionment_TypeManual input → dimAccount, VersionTags each account Direct or Indirect
CST_Input_Driver_Pct_Share (%)Formula + overrideAccount, CostCenter, CostPool, Version, MonthReads driver % from CST_TXN_Account_Pool_Pct
CST_Input_Direct_Pct_Share (%)Formula + overrideAccount, CostCenter, ServiceLine, Version, MonthReads direct split % from CST_TXN_Direct_Account_SL_Pct
CST_Calc_Account_to_Pool ($)FormulaAccount, CostCenter, CostPool, Version, MonthIndirect amount × driver % ÷ 100
CST_Calc_Stage1_Pool_AmountFormulaCostCenter, CostPool, Version, MonthAggregates account-to-pool using the REMOVE modifier
Formula pattern

SELECT filters to Indirect accounts only before multiplying by driver percentage. REMOVE then collapses the CST_Account dimension to produce a clean per-pool total.

RC01 — total input overhead must equal total Stage 1 pool output
1b

Pool → Pool · reciprocal

Reciprocal settlement between cost pools
Board Stage 1b — Pool to Pool

Some pools serve other pools — an IT pool serving an HR pool, say. This stage performs reciprocal settlement: each pool redistributes a portion of its cost to the pools it serves, then receives back the cost of the pools that served it. It is an iterative calculation that converges to a stable allocation.

How it works in Pigment

Because Pigment has no native circular or iterative formula, settlement is computed externally — via ETL / webhook — and written back into CST_TXN_P2P_Final_Settlement. Pigment simply reads the settled amounts.

CST_Input_P2P_Driver_TypeManual input → dimCostPool, VersionAssigns the driver used for P2P splits
CST_Input_P2P_Is_ActiveFormula + overrideCostPool, CostPoolDest, Version, MonthFlags active pairs from CST_TXN_P2P_Active_Pairs
CST_Input_CC_Driver_ValueFormula + overrideCostCenter, Driver, Version, MonthDriver volumes from CST_TXN_P2P_Driver_Volumes

Current limitation

The recursion webhook that triggers iterative settlement is not yet active. Until it is, users trigger settlement manually.

Formula pattern · the sentinel in action

The settled pool amount is stored at the Activity = "Input" coordinate. The formula uses SELECT: CST_Activity.Name = "Input" to isolate that total, then multiplies it by the normalised driver percentage across all real activities.

06 — Control

The control & reconciliation layer

All reconciliation logic is consolidated into a set of CST_Ctrl_ metrics structured by the CST_ReconciliationCheck dimension. This gives one unified view across all 13 checks — RC01–RC06 plus VAL00–VAL05.

MetricRole
CST_Ctrl_Input_Total ($)Sum of the RC and VAL input metrics — total expected input per check
CST_Ctrl_Input_RC ($)Handles RC-type checks only (isolated — see the design note)
CST_Ctrl_Input_VAL ($)Handles VAL-type checks only (same reason)
CST_Ctrl_Output_Total ($)Total expected output per check
CST_Ctrl_Status"PASS" if ABS(Input − Output) ≤ $0.01, else "FAIL"
CST_Ctrl_MessageHuman-readable variance — "Variance: $X"
CST_Ctrl_Total_Overhead ($)Grand total overhead across all pools
CST_Ctrl_Total_Direct ($)Grand total direct cost
CST_Ctrl_Total_Overhead_by_ActivityDest ($)Overhead by ActivityDest — a wrapper enabling SELECT on CST_ActivityDest downstream

Design note — the IF-formula scope trap

This one is worth internalising, because it produces silent wrong numbers rather than an error.

The behaviour

CST_Ctrl_Input_Total ($) began as a single metric with one IF formula branching across RC and VAL check types. When an IF references dimensions in any branch, Pigment eagerly expands the metric's calculation scope to include all referenced dimensions — even for rows where that branch is never evaluated. Aggregated back to a dimension the metric didn't natively hold, this produced silent value inflation.

The fix: split the logic into two metrics — CST_Ctrl_Input_RC and CST_Ctrl_Input_VAL — each carrying only its own check type and dimensional context, then sum them in CST_Ctrl_Input_Total. Isolating scope per metric eliminates the inflation.

07 — Interface

Boards & navigation

The interface is seven Pigment boards in a dedicated folder. Navigation is bidirectional — users move forward through the process from Home, and return home from any stage.

CST Home — Cost Apportionment  [HOME]
    │
    ├── Stage 1  — Account to Pool          (RC01)
    ├── Stage 1b — Pool to Pool             (RC02)
    ├── Stage 2  — Pool to Activity         (RC03)
    ├── Stage 2b — Activity to Activity     (RC04)
    ├── Stage 3  — Activity to Service Line (RC05, RC06)
    │
    └── Validation Gate  [final sign-off]

Home carries a process overview, a warning banner that the recursion webhook is not yet active, navigation cards to each stage and the Validation Gate, and descriptive text for each step.

Each stage board holds configuration tables (driver type, active-pair flags, driver volumes), a debug/review table of the calculated intermediates, a PASS/FAIL reconciliation result, and a back-to-Home button.

The Validation Gate consolidates all RC and VAL checks into a single pre-flight dashboard — the launch point for the final apportionment run.
