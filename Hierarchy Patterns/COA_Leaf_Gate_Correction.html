Snyk COA · Correction Note

Mapping Properties — Leaf Gate Correction

Four edits to the A.1–A.4 mapping properties. No change to any number on screen.

Before you start — nothing is broken

The model produces correct results today. This correction makes the formulas say what they currently do by accident. If Saturday runs short, this can wait — but it should not ship to handover without it.

1. What is wrong

Each mapping property has a branch that says, in effect: “if you sit at this level’s depth, map to yourself.”

// A.2, BS branch — as currently built
IF('A.0 Chart of Accounts Flat'.Depth = 3,
    ITEM('A.0 Chart of Accounts Flat'.Account_Code, 'A.2 Level 2'.Account_Code),
    ...

But two different kinds of Element sit at Depth 3 in the BS:

  • A real L2 node — has children. It belongs in A.2 Level 2 and should map to itself.
  • A ragged leaf — no children, its branch simply stops here. It belongs in A.0 only, and should be blank at A.2.

Depth alone cannot tell them apart. Both are Depth 3. Only the leaf flag separates them — and the formula does not check it.

2. Why it currently works anyway

The A.2 Level 2 import filters on L2 = TRUE, and the L2 boolean contains ISBLANK(L0) — so leaves were never loaded into the level list.

A ragged leaf therefore self-maps, ITEM searches A.2 Level 2 for its Account_Code, finds nothing, and returns blank.

Right answer, wrong reason

The blank arrives because a lookup silently failed, not because the formula decided the leaf has no L2. Three consequences:

  • Anyone reading the formula sees leaves self-mapping into level lists. The intent is invisible.
  • If a leaf were ever loaded into a level list — a changed import filter, a data fix — the mapping would begin returning wrong Elements, silently.
  • The placement booleans gate on the leaf flag; the mapping properties do not. The two disagree about the same question.

3. The rule

Every self-map branch needs a leaf gate

At a level’s own depth: if not a leaf → map to yourself. If a leaf → BLANK.
Hop branches (walking up to an ancestor) need no gate — a leaf walking upward resolves correctly.

Only one branch per property changes: the Depth = <this level’s depth> branch, in each of the BS and PL sections.

PropertySelf-map branch (BS)Self-map branch (PL)
A.1 Level 1Depth = 4Depth = 2
A.2 Level 2Depth = 3Depth = 1
A.3 Level 3Depth = 2Depth = 0 (root)
A.4 Level 4Depth = 1— (no PL clause)

The PL root at Depth 0 is never a leaf, so A.3’s PL self-map branch is harmless either way — gate it anyway for consistency.

4. Corrected formulas

Gated branches are the ones wrapped in IF(ISBLANK(…L0), …, BLANK). Everything else is unchanged.

A.1 Level 1 — type: dimension → A.1 Level 1

// A.1 Level 1 — the Level 1 Element this Element rolls into.
// L1 sits at Depth 4 (BS, frame 5) and Depth 2 (PL, frame 3).
// Self-map branches are gated on the leaf flag: a ragged leaf at the level's
// own depth has no L1 member and must return BLANK.
IF('A.0 Chart of Accounts Flat'.Statement = "BS",
    IF('A.0 Chart of Accounts Flat'.Depth = 5,
        ITEM('A.0 Chart of Accounts Flat'.Element_Parent.Account_Code, 'A.1 Level 1'.Account_Code),
    IF('A.0 Chart of Accounts Flat'.Depth = 4,
        IF(ISBLANK('A.0 Chart of Accounts Flat'.L0),
            ITEM('A.0 Chart of Accounts Flat'.Account_Code, 'A.1 Level 1'.Account_Code),
            BLANK),
        BLANK)),
    IF('A.0 Chart of Accounts Flat'.Depth = 3,
        ITEM('A.0 Chart of Accounts Flat'.Element_Parent.Account_Code, 'A.1 Level 1'.Account_Code),
    IF('A.0 Chart of Accounts Flat'.Depth = 2,

5. Verify after the edit

Nothing on screen should change. Confirm that:

  • Each mapping column shows the same values as before for a handful of known Elements — a deep leaf, a shallow leaf, an intermediate node, a root.
  • A ragged leaf sitting at a level’s own depth is blank at that level, as it was.
  • Element counts in A.1–A.4 are unchanged (the imports are untouched).

Two open items — not part of this correction

The build rule is not written down. These level dimensions carry blanks. Any metric aggregated BY a level will silently drop the blank-mapped Elements — no error, totals short. A view grouped by the same property retains them and totals correctly. Before handover, add to the design notes:

“Level dimensions carry blanks where a branch skips a level. Any metric aggregated BY a level will silently drop those Elements. Level-grain metrics require a complete member set (a Calc helper dimension) before use.”

Level numbering is bottom-up. L1 = one above leaf, so the same label refers to different tiers in BS and PL, and the statement roots sit at different level numbers. Deepening the hierarchy shifts every label, because new levels appear at the leaf end. Worth one factual line in the design notes; not worth reworking.

6. Why the hop branches need no gate

A leaf walking upward resolves correctly regardless of leaf-ness — its ancestor at the target depth is a real node, and real nodes are in the level list. The ambiguity exists only at a level’s own depth, where a real node and a ragged leaf are indistinguishable by Depth alone.

This is the same reason the balanced pattern needs no leaf detection at all: “at your own depth, return yourself” is correct for a real node (legitimately) and for a leaf (as promotion). Only a blank-based resolver has to tell them apart, and must therefore gate every level on a downward child-count.
