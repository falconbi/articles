Ragged Hierarchy Pattern in Pigment — Lean Single-Master

A Modelling & Implementation Reference

Version: 1.0 Author: James Lovering Last Updated: July 2026 Applies to: Simple, low-governance ragged hierarchies where a single maintainer controls the model.
Start here first: Ragged Hierarchies in Pigment — Concepts & Pattern Selection, which explains the failure mode and the choice between this pattern and the Balanced Levels companion.

Read this first. This is the lean pattern — optimised for maintenance simplicity and clean UI, at the cost of a narrow but real calculation hazard that must be managed by discipline. For any model where many hands will touch it, governance is weak, or level-grain calculations are known to be needed, use the companion document Ragged Hierarchy Pattern — Balanced Levels, which is loss-proof by construction. Section 1 below states exactly when each is appropriate.

Disclaimer — reference, not warranty. This is a practitioner design reference, not a guarantee of fitness for any particular model. Validate every formula and behaviour against your own environment and requirements before relying on it in production. Specific things to re-check:

• Platform behaviour can change. Formula and aggregation behaviour described here (including the BY blank-drop and the BY COUNT child-count idiom) was observed on Pigment as of July 2026; re-test after platform updates, as vendor behaviour may change without notice and is, in places, not officially documented.
• Model-confirmed vs logic-verified. Some formulas here are confirmed in a working model and others are logic-verified pending in-model validation — see the confirmation-status note in Section 3.2. Do not treat the two as equivalent.
• Assumptions. The pattern assumes a unique Name/Code per item, a non-blank Internal_ID on every row, and a clean single-parent tree. Confirm these hold in your data.
• No professional advice. Nothing here is legal, financial, or accounting advice. Modelling choices with reporting or compliance consequences are yours to validate.

Provided as-is, without warranty of any kind. You are responsible for testing and validating to your own needs.

Naming note: formulas use Cost Centre Load, Cost Centre L1–L7, and Cost Centre (the position-free leaf) as the worked example.

1. When to Use This Pattern

Both patterns solve the same ragged-hierarchy problem and rest on the same underlying necessity — every calculation grain needs a complete set of members behind it. They differ in when that completeness is built and where the members live.

Concern Balanced (companion doc) Lean (this doc)
Calculation safety Every level, by construction Leaf / L1 / views safe as-built; level-grain metrics require a helper (Section 5)
When safety is paid for Up front (promotion, per-level imports) On demand (helper built when a consumer appears)
Level list contents Real + promoted members Real members only (display); synthetics confined to a Calc helper
View / UI appearance Shows promoted / "No Rollup" rows Clean — real members only; ragged leaves nest under their BU
Hierarchy changes Touch multiple lists via imports Touch the Load List, re-import projections
Governance burden None ongoing — structure enforces safety One rule, enforced forever: level-grain metrics → Calc helper
Fails how, if misused Hard to misuse Silently, if the rule is breached

Use the lean pattern when: the model has a single knowledgeable maintainer, level-grain calculations are absent or rare, and clean member-only reporting is a priority. Use the balanced pattern instead when: many people build in the model, governance is weak, or department/BU-grain rules and budgets are expected — in which case building completeness up front costs no more and removes the hazard entirely.

2. Architecture

The lean pattern treats the Load List as the permanent master. Operational dimensions are thin projections of it, populated by import (see the separation rule below).

Block Role
Cost Centre Load Master — every node, all levels; source of truth for all hierarchy logic
Cost Centre Operational leaf dimension for all metrics — leaf items only (position-free name; Section 6)
Cost Centre L1, L2, … Thin display lists — real members of each level only; no synthetic or promoted members

The Load List carries the same machinery detailed in the companion document — Parent_Internal_ID, Depth, the corrected Depth Exceeded (see Section 3), and the Level 1–Level 7 resolvers. The one deliberate difference from the balanced pattern:

Level resolvers return BLANK where no ancestor exists at that level. A Depth-1 ragged leaf has Level 2 = BLANK. There is no promotion — this is what keeps the display lists to real members only, and it is also the source of the calculation hazard in Section 4.

Seven levels are provisioned from day one and parked until needed. The Level 1–Level 7 resolvers exist and compute immediately, whatever depth today's data reaches. Unused deeper levels sit dormant at no cost — a hierarchy only 2–3 deep simply leaves L4–L7 idle, and the unused stub properties can be set to hidden to keep the property list uncluttered (their activation instructions live in their own formula comments, so nothing is lost by hiding them). The ceiling is free headroom, not future work; you build no levels you don't need and retro-fit none you do.

Two distinct numbers, easily conflated. The frame is the height the hierarchy actually has — a property of the tree. The ceiling is the depth the resolvers support — a property of this template. A hierarchy with a frame of 3 sits comfortably under a ceiling of 7. See Concepts §1.

The Separation Rule (Paramount — applies to this pattern too)

Production lists are populated by import from the Load List — never implemented as List Subsets of it, and never live-derived by formula from it.

Loads must be checkable before they reach the model. An import is a gate: refresh the Load List, inspect and validate it, then deliberately push it into the operational dimensions that models and apps use. A subset or a formula-derived link removes that gate and makes production a live reflection of staging.

Consequence for this pattern — read carefully. A "self-maintaining" variant is often proposed for the lean pattern, in which the operational dimensions' parent-link properties are formulas that resolve from the Load List at runtime (the double-ITEM bridge). It recalculates on every reload with no import step — but it is exactly the live-coupling the separation rule forbids: the Load List becomes a permanent runtime dependency, and any load defect reaches calculations with no checkpoint. This pattern therefore uses import-populated projections, the same as the balanced pattern. The maintenance saving of the lean pattern comes from the single flat master import and the pre-written resolvers — not from live-derived links, which are deliberately rejected.

3. Corrected Depth Exceeded (Applies to Both Patterns)

The overflow warning must detect depth directly, not from the capped Depth value.

IMPORTANT — do NOT use IF(Depth > 6, ...). The Depth property is itself capped at 6 by its IFDEFINED chain, so Depth > 6 is unreachable dead code: an item 8 levels deep still reports Depth = 6 and the warning never fires.

// Corrected overflow detector — tests a 7th parent hop directly.
// If a 7th hop resolves, the item exceeds the 7-level ceiling.
// To extend the ceiling to 8 levels: add one more Parent_Internal_ID hop to this chain.
IFDEFINED(
    'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
    "⚠️ Depth > 6 — hierarchy exceeds 7 levels. Extend Depth, Level properties, and parent hops.",
    BLANK
)

3.1 Has_Children Flag — Why the Blank Pattern Needs It

Before the Level resolvers, the blank pattern needs one extra property the balanced pattern does not: a Has_Children flag. Here is why.

At any given depth, an item can be one of two very different things, and depth alone cannot tell them apart. Under one BU root, at Depth 1, you may have both a real department (which itself has teams below it) and a bare cost centre (a ragged leaf with nothing below it):

Marketing                Depth 0  (BU root)
├── Digital Team         Depth 1  ← has children → a real L2 department
│   ├── Digital AMER     Depth 2  (leaf)
│   └── Digital EMEA     Depth 2  (leaf)
├── AMER Marketing       Depth 1  ← no children → a ragged leaf
└── Brand Development    Depth 1  ← no children → a ragged leaf

All three middle items are Depth 1, but at Level 2 they must resolve differently: Digital Team is a genuine L2 department and should return itself; AMER Marketing and Brand Development are ragged leaves with no L2 role and should return BLANK. The distinguishing fact is not depth — it is whether the item has children.

This ambiguity is unique to the blank pattern. The balanced pattern sidesteps it entirely: under promotion, “Depth 1 → itself at Level 2” is correct for both — a real department returns itself, and a ragged leaf returning itself is the intended promotion. Only the blank pattern must actively distinguish the two, which is why only it needs the Has_Children flag. A genuine simplicity point in the balanced pattern’s favour.

Has_Children — reverse traversal via BY COUNT

Detecting children means aggregating from child items back to their parent — a reverse traversal. Property chains only walk up (child → parent), never down, so you cannot ask an item to list its children directly. The correct Pigment idiom is BY COUNT: group all items by their Parent_Internal_ID and write each parent’s child-count back onto it.

// Has_Children — TRUE if any item names this item as its Parent_Internal_ID.
// Internal_ID is any non-blank value on every item, used purely as something to count.
// BY COUNT groups all items by their parent, tallying children onto each parent slot.
// > 0 is TRUE for parents; leaves are never counted, so they read as BLANK/FALSE.
'Cost Centre Load'.Internal_ID [BY COUNT: 'Cost Centre Load'.Parent_Internal_ID] > 0
// Is_Leaf — the inverse. NOT on a blank (unset leaf) returns TRUE, so no IFDEFINED guard is needed.
NOT 'Cost Centre Load'.'Has_Children'

Verify NOT against a blank in your own model. The form above was confirmed against a Has_Children flag derived from the BY COUNT metric. A separate build (July 2026) found NOT failing to evaluate against a Boolean list property holding TRUE/blank, where ISBLANK(…) worked correctly:

// Worked where NOT did not, on a TRUE/blank Boolean list property:
ISBLANK('Cost Centre Load'.'Is_Leaf')

The two observations may differ because one flag is metric-derived and the other is a plain list property, or one may have been misdiagnosed. Both are unresolved as of this version. Test NOT on your own blanks before relying on it; prefer ISBLANK() where the operand is a TRUE/blank list property.

MATCH — the same signal as a list property

BY COUNT requires a metric object. MATCH answers the identical downward question as a plain list property, at the cost of a text Parent_Code to search against:

// Is_Leaf — nobody references this item's Code as their Parent_Code.
IF(ISBLANK(MATCH('Cost Centre Load'.Code, 'Cost Centre Load'.Parent_Code)), TRUE)

MATCH is the cleaner default for small and moderate lists — one property, no metric object. BY COUNT is the proven idiom at volume: it groups once, where MATCH searches a column per row. MATCH has not been benchmarked on large dimensions; test before committing on a list of tens of thousands of items.

Note: unlike the balanced pattern, leaf detection here is load-bearing. Every Level resolver above Level 1 gates on it to distinguish a real node from a ragged leaf at the same Depth. It is not an optional QA layer.

Empirically confirmed: for one root with three Depth-1 children — one with grandchildren, two without — the flag reads TRUE for the department and FALSE for the two leaves, because BY COUNT only places a value on items that actually appear as some other item’s parent.

3.2 Level Resolver Formulas — Blank Form, Has_Children-Aware (all 7 levels)

These are NOT the balanced pattern’s resolvers. Here a resolver returns BLANK where the item has no true ancestor at that level — there is no promotion. At the target depth, an item returns itself only if it has children (a real node at that level); a leaf at the same depth returns BLANK. If you copy the balanced pattern’s promoting resolvers by mistake you will have built the balanced pattern, not this one. Level 1 is identical in both patterns (roots are never ragged and always return themselves); Level 2 onward carry the Has_Children gate.

The rule for Level N (target depth = N−1): shallower than the target → BLANK; exactly at the target depth → itself if Has_Children, else BLANK; deeper than the target → walk up (Depth − target) parent hops. All seven are written out below.

Model-name mapping. These formulas use the document’s naming, Cost Centre Load. The working Pigment model uses Cost Center Load Option 3 (US spelling, with the Option 3 suffix). The two are the same list — substitute the name your model actually uses when copying in either direction.

Confirmation status. Parent_Internal_ID, Depth, Element Level and Level 1 are confirmed working in the model. The Has_Children flag and the Level 2–7 gates are logic-verified here (every level checked against the depth/hop rule) and follow the Pigment-AI-provided BY COUNT idiom; validate the BY COUNT result and one mixed-depth case (a Depth-1 department resolving to itself, a Depth-1 leaf to BLANK) in the model before relying on them in production calculations.

Level 1 — root ancestor (no Has_Children gate; a root is always the L1 ancestor)

// Level 1 ancestor. Returns the root (Depth 0) for any item. Identical to the balanced pattern.
IF('Cost Centre Load'.Depth == 0, ITEM('Cost Centre Load'.Name, 'Cost Centre Load'.Name),
IF('Cost Centre Load'.Depth == 1, 'Cost Centre Load'.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 2, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 3, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 4, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 5, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 6, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
BLANK )))))))

Level 2 — target Depth 1. A Depth-1 item returns itself only if it has children; a ragged leaf returns BLANK.

// Level 2 — Has_Children-aware.
// Depth 0: BLANK   Depth 1 + Has_Children: itself   Depth 1 + leaf: BLANK
// Depth 2: 1 hop   Depth 3: 2   Depth 4: 3   Depth 5: 4   Depth 6: 5
IF('Cost Centre Load'.Depth == 0, BLANK,
IF('Cost Centre Load'.Depth == 1,
    IF('Cost Centre Load'.'Has_Children', ITEM('Cost Centre Load'.Name, 'Cost Centre Load'.Name), BLANK),
IF('Cost Centre Load'.Depth == 2, 'Cost Centre Load'.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 3, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 4, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 5, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 6, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
BLANK )))))))

Level 3 — target Depth 2.

// Level 3 — Has_Children-aware.
// Depth 0-1: BLANK   Depth 2 + Has_Children: itself   Depth 2 + leaf: BLANK
// Depth 3: 1 hop   Depth 4: 2   Depth 5: 3   Depth 6: 4
IF('Cost Centre Load'.Depth <= 1, BLANK,
IF('Cost Centre Load'.Depth == 2,
    IF('Cost Centre Load'.'Has_Children', ITEM('Cost Centre Load'.Name, 'Cost Centre Load'.Name), BLANK),
IF('Cost Centre Load'.Depth == 3, 'Cost Centre Load'.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 4, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 5, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 6, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
BLANK ))))))

Level 4 — target Depth 3.

// Level 4 — Has_Children-aware.
IF('Cost Centre Load'.Depth <= 2, BLANK,
IF('Cost Centre Load'.Depth == 3,
    IF('Cost Centre Load'.'Has_Children', ITEM('Cost Centre Load'.Name, 'Cost Centre Load'.Name), BLANK),
IF('Cost Centre Load'.Depth == 4, 'Cost Centre Load'.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 5, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 6, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
BLANK )))))

Level 5 — target Depth 4.

// Level 5 — Has_Children-aware.
IF('Cost Centre Load'.Depth <= 3, BLANK,
IF('Cost Centre Load'.Depth == 4,
    IF('Cost Centre Load'.'Has_Children', ITEM('Cost Centre Load'.Name, 'Cost Centre Load'.Name), BLANK),
IF('Cost Centre Load'.Depth == 5, 'Cost Centre Load'.Parent_Internal_ID,
IF('Cost Centre Load'.Depth == 6, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID,
BLANK ))))

Level 6 — target Depth 5.

// Level 6 — Has_Children-aware.
IF('Cost Centre Load'.Depth <= 4, BLANK,
IF('Cost Centre Load'.Depth == 5,
    IF('Cost Centre Load'.'Has_Children', ITEM('Cost Centre Load'.Name, 'Cost Centre Load'.Name), BLANK),
IF('Cost Centre Load'.Depth == 6, 'Cost Centre Load'.Parent_Internal_ID,
BLANK )))

Level 7 — target Depth 6 (deepest supported; no deeper hops possible).

// Level 7 — Has_Children-aware.
IF('Cost Centre Load'.Depth <= 5, BLANK,
IF('Cost Centre Load'.Depth == 6,
    IF('Cost Centre Load'.'Has_Children', ITEM('Cost Centre Load'.Name, 'Cost Centre Load'.Name), BLANK),
BLANK ))

The shape is mechanical: for Level N, everything shallower than Depth N−1 is BLANK; an item exactly at Depth N−1 returns itself if it has children (a real node at that level) and BLANK otherwise (a ragged leaf); each deeper depth adds one more Parent_Internal_ID hop. The Has_Children gate at the target depth is the whole difference from a naive depth-only resolver — it is what correctly separates a real internal node from a ragged leaf sitting at the same depth.

4. What Is Safe, and the Known Edge

4.1 Safe As-Built

  • All leaf-grain calculation — metrics live on Cost Centre; blanks never enter.
  • L1 references — every item has a root; the property is always populated.
  • Node-keyed assumptions with inheritance — rates and drivers entered against any node in the master list; leaves resolve by nearest-ancestor climb: IFBLANK(rate[self], IFBLANK(rate[parent], …)). One formula handles both blank kinds (no ancestor at a level; no entry at a node), and this is a genuinely good pattern in its own right (enter one BU rate, override per department only where it differs).
  • Driver-based apportionment — drivers held at leaf; pools spread across leaves; levels never referenced.
  • All view-based reporting, including charts — charts are configured from views; views group by properties and retain blank-mapped items under a blank header with complete totals (empirically: view total 28 where the equivalent BY metric showed 13).

4.2 The Known Edge

Two families cannot be handled by build style alone:

  1. Metrics dimensioned by a skippable level. Any [BY: …'Level N'] into a level-dimensioned metric silently drops every item blank at that level — totals short, no error.

Empirically confirmed (July 2026): a four-item fixture (two items with a populated intermediate property, two blank, values totalling 28) aggregated via [BY: ...] to that property returned 13. The two blank items vanished with no error and no null row. The control (fully-populated property) returned 28; a view over the same blanks also returned 28.

Fetches from such metrics back down to ragged leaves return BLANK and propagate through arithmetic (a blank rate quietly zeroes an allocation).

  1. Member-shaped uses. Level selectors on boards (blank is not selectable — ragged leaves become unreachable), subtotal rows in calculated output (no member, no row), and complement/set filters (L2 <> X — blank membership is undocumented and nondeterministic).

The rule: a level is calculation-safe iff no item skips it. In a ragged hierarchy the skippable levels are exactly where the hazard lives, and the set of skippable levels grows as the hierarchy deepens.

4.3 Handling Isolated Cases

Occasional edge cases can be patched per-formula — legitimate rule logic:

// Fallback up the chain: department rate, else BU rate.
'Cost by Leaf' * IFBLANK('Rate by L2'[BY: 'Cost Centre'.'Cost Centre L2'],
                         'Rate by L1'[BY: 'Cost Centre'.'Cost Centre L1'])

This is fine for one-off rules. But the safety now lives in every such formula, applied by every author, forever — a discipline tax with silent failure on any lapse. Recurring or member-shaped cases should instead be fixed structurally:

5. The Helper Bridge — Deferred Completeness

When a genuine level-grain consumer appears — a department-held budget to variance against, an L2 selector requirement — add a helper Calc dimension rather than patching formulas one by one.

  • Cost Centre L2 Calc = all real L2 members plus one synthetic member per root ("Marketing – Direct", "G&A – Direct", …), each parented to its root. Per-root synthetics are forced: a single global "Undefined" member cannot carry a coherent L1 parent link across roots — the 67 ragged leaves in the worked data span six BUs.
  • One mapping property on Cost Centre, coalesced once at source so it is never blank:
// 'L2 Calc' on the leaf — never blank by construction.
// The synthetic per-root member gives ragged leaves a real home at L2 grain.
IFBLANK( ITEM('Cost Centre'.'Cost Centre L2'.Name, 'Cost Centre L2 Calc'.Name),
         <this leaf's per-root synthetic in L2 Calc> )

Every level-grain calculation then uses the helper: [BY: 'Cost Centre'.'L2 Calc'], rates dimensioned by Cost Centre L2 Calc, filters and selectors on L2 Calc. Because the helper is a complete member set, it clears the member-shaped cases too: complement filters become deterministic, every leaf is selector-reachable, every subtotal row exists. The display list stays pure, so views keep their clean real-members-only appearance.

Everything the helper needs already exists in the Load List resolvers, so it is buildable at any time — an import and one property, zero rework of existing structures. This is why the lean pattern can defer completeness: you pay for it only if and when a level-grain consumer actually appears.

The Non-Deferrable Build Rule

Any metric dimensioned by a level uses the Calc helper list, never the display property.

A rate table or aggregation accidentally built on the display property quietly reinstates the blank-drop for ragged leaves. This single documented rule is the lean pattern's entire governance burden — much smaller than "handle blanks in every formula," but it must be enforced, and it fails silently if breached. This is the reason the lean pattern is recommended only for models with disciplined, knowledgeable maintenance.

Note: the helper is a calc structure. Calculation outputs at L2 grain (e.g. a variance report) will show the synthetic "– Direct" rows — correctly, because that is where the ragged leaves' values live. You move where the synthetics appear (calc outputs, not display rollups), not whether they are ever seen. If members-only output is required everywhere, that requirement is itself a reason to prefer the balanced pattern.

6. Naming Rule

The metric-facing dimension carries a position-free name — Cost Centre, never Cost Centre L3. Numbered names are stable only for positions that don't move, and deepening moves the leaf. The upper levels keep top-anchored numbers (Cost Centre L1, L2, …). With the position-free name, deepening costs two actions — create the new intermediate level list, switch the leaf import's source from Level N to Level N+1 — and zero rule changes, because rules only ever reference 'Cost Centre' and the stable upper levels.

True-Leaf Membership — A Deepening Caution

Unlike the balanced pattern (where promotion makes leaf membership additive), the lean pattern defines the leaf dimension as true leaves only. This has a consequence when a leaf gains children — see Section 7.

7. True-Leaf Instability on Deepening (Design Limitation)

This is a consequence of the lean pattern's definition of "leaf" — not a formula error, and no formula can fix it. It is the primary structural cost of the lean pattern when the hierarchy may grow, and the main reason to prefer the balanced pattern when future deepening is plausible.

7.1 What Happens When a Depth-2 Leaf Gains a Child

Demonstrated empirically: adding "Accounting - AP" as a child of Accounting (a Depth-2 leaf in the leaf dimension) produced the following state:

Question Observed result
Does Accounting still appear in the leaf dimension? YES — Pigment made no automatic change
Is "Accounting - AP" automatically added to the leaf dimension? NO — it only exists in the Load List
Is Accounting's metric data at risk? YES — removing it from the leaf dimension orphans all stored values

7.2 Why This Is Structural

The lean pattern defines leaves implicitly: the leaf dimension is a curated list. The Load List's Level formulas have no knowledge of who is in the leaf dimension and cannot self-correct when the hierarchy grows. A node that was a leaf yesterday and has a child today is still, incorrectly, a member of the leaf dimension — and its own value now risks being double-counted against its new children's rollup, or lost if it is removed.

The balanced pattern does not have this problem: promotion makes a former leaf its own descendant at deeper levels, so it remains a valid, postable member of the leaf grain automatically — membership is additive, and existing data survives.

7.3 Required Migration Procedure (No Built-In Automation)

When a leaf gains children in production:

  1. Identify impact — list every metric dimensioned by the leaf dimension.
  2. Snapshot data — export all values stored against the former leaf.
  3. Add new items — add the sub-items to the Load List (with correct Parent_Internal_ID) and ensure they flow into the leaf dimension on the next import.
  4. Remap data — re-enter or import historical data against the new sub-items.
  5. Resolve the former leaf — decide whether it remains a postable member or is removed; if removed, its stored values are lost unless remapped in step 4.
  6. Human decision required — how historical data is redistributed to children cannot be automated; it requires a business decision per affected metric.

This migration event is the primary structural cost of the lean pattern on deepening, and the main reason to choose the balanced pattern when the hierarchy may grow.

8. Maintenance Profile & Summary

What the lean pattern buys: hierarchy changes are made in the Load List — one flat import with Parent IDs, resolvers pre-written to 7 levels, no per-level source files. Display lists show only real organisational members, so everyday reporting and boards are clean, with ragged leaves nesting naturally under their BU.

What it costs: a narrow but silent calculation hazard at any skippable level, managed by the Section 5 build rule (level-grain metrics → Calc helper) and, on deepening, the Section 7 true-leaf migration. Both are manageable by a disciplined single maintainer; both are traps for a team without that discipline.

The honest bottom line: the lean and balanced patterns are the same architecture with completeness built lazily versus eagerly. The lean pattern is a legitimate choice for simple, well-governed models where clean member-only display matters. Where data-loss safety must be guaranteed regardless of who touches the model next, use the balanced pattern.
