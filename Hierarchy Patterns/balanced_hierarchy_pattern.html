Ragged Hierarchy Pattern in Pigment — Balanced Levels

A Modelling & Implementation Reference

Version: 1.0 Author: James Lovering Last Updated: July 2026 Applies to: Any dimension list with a parent-child relationship of variable depth.
Start here first: Ragged Hierarchies in Pigment — Concepts & Pattern Selection, which explains the failure mode and the choice between this pattern and the Lean Single-Master companion.

At a Glance
Problem Ragged hierarchies break level-based grouping: shallow branches have no ancestor at deeper levels, so items silently vanish from grouped reports and aggregations.
Solution A flat Load List with pre-computed Level 1–Level N ancestor properties, feeding one curated dimension list per level by import. Shallow leaves promote themselves at deeper levels so every node exists at every level — no blanks anywhere a calculation can meet one.
Safety Every level is calculation-safe by construction. BY aggregation, fetches, filters, and selectors all resolve for every item, because no level property is ever blank.
Ceiling Handles 7 levels (Depth 0–6) out of the box — resolvers pre-built to all 7, with unused levels parked dormant until ever needed. Extending beyond 7 is mechanical (Section 6).
Key invariant Ancestor at Level N for a node at Depth D = D − (N−1) parent hops. If D ≤ N−1, the node returns itself (at-level or leaf promotion).
Separation rule Production lists are import-populated from the Load List, never List Subsets or live-derived. The import is the gate: check the Load, then update the model. See Section 2.

1. What Is a Ragged Hierarchy?

A ragged hierarchy (also called an unbalanced or variable-depth hierarchy) is one where different branches of the tree reach different depths.

In a balanced hierarchy every leaf sits at the same depth. In a ragged one, some leaves are shallower than others. This breaks naive level-based reporting: you cannot simply "group by Level 3" because some items (Finance) have no Level 3 ancestor. Without intervention, those items either disappear from the grouped view or fall into an "unknown" bucket — both are silent data-loss failures.

This pattern eliminates the failure at its root: it ensures every node has a defined ancestor at every level, so no blank ever reaches a calculation.

2. The Two-Tier Architecture

The pattern separates concerns into two tiers, and the separation is enforced by import, not by live derivation.

Tier 1 — The Load List

A single flat dimension list holding every node in the hierarchy (roots, branches, and leaves) with a self-referential Parent_Internal_ID property. This is the staging layer: imported from source data, never used directly in metrics. All hierarchy intelligence (depth detection, ancestor walkout, level joins) is computed here as properties.

Naming convention: [Entity] Load (e.g., Cost Centre Load)

Tier 2 — Curated Dimension Lists

One standalone dimension list per level, plus the leaf dimension used by metrics, each containing only the items belonging to it, populated by extracting values from the Load List.

  • The upper-level lists are named by position from the top: Cost Centre L1, Cost Centre L2, … These are stable names because roots are always L1, the tier below always L2, and so on — deepening adds levels below, never above.
  • The metric-facing leaf dimension is named Cost Centre — position-free, permanent. It is never called Cost Centre L3 even though today's leaves happen to sit at level 3; see Section 7.3 for why the number must not appear in the leaf's name.

Each list carries a dimension property linking it to its parent list (Cost Centre → Cost Centre L2 → Cost Centre L1), so the full hierarchy is available in any view of a leaf-dimensioned metric by grouping on those linked properties — no additional metric dimensions required. Structure and import mechanics are covered in Section 4.

Leaf Detection — Two Idioms, One a Metric

Leaf-ness is a downward fact: “does anything name me as its parent?” It cannot be derived from the hierarchy’s upward structure at all (see 3.6). Any classification built on ancestors alone is wrong for ragged data. Two idioms answer the downward question.

MATCH — a List Property. Searches the parent column for this item’s key. Self-contained on the Load List, no extra object. Requires a text Parent_Code (derived as Parent.Code), because MATCH cannot compare a dimension reference against text.

// Is_Leaf — nobody references this item's Code as their Parent_Code.
IF(ISBLANK(MATCH('Cost Centre Load'.Code, 'Cost Centre Load'.Parent_Code)), TRUE)

BY COUNT — a metric. Groups all items by their parent and tallies children onto each parent slot. BY COUNT cannot be expressed as a List Property — it must be a metric dimensioned by the Load List. So the Cost Centre Child Count metric sits beside the Load List, reading from it and feeding back onto its classification flags. Disposable and staging-only, never a metric the model plans against.

Which to use. MATCH is the cleaner default for small and moderate lists — one property, no metric object. BY COUNT is the proven idiom at volume: it groups once, where MATCH searches a column per row. MATCH has not been benchmarked on large dimensions; test before committing on a list of tens of thousands of items.

Note on necessity. In the balanced pattern the Level resolvers never consult a leaf flag — “at your own depth, return yourself” is correct for a real node (legitimately) and for a leaf (as promotion). Leaf detection is therefore not load-bearing here; it supports the optional QA classification layer of Section 3.6 and nothing else. It is load-bearing in the lean pattern, whose resolvers must gate every level on it. This asymmetry is one of the clearest practical differences between the two.

Full definitions and dependencies are in Sections 3.6 and 5.1.

The Separation Rule (Paramount)

Production lists are populated by import from the Load List — never implemented as List Subsets of it, and never live-derived by formula from it.

The reason is control. Loads must be checked before they reach the model: an import gives an explicit gate where the Load List can be refreshed, inspected, and validated, and only then deliberately pushed into the curated lists that models and apps depend on. A List Subset (or a formula-derived link) dissolves that gate — it makes the production dimensions a live reflection of the Load List, so any load defect, mid-load state, or source-format change propagates instantly into calculations with no checkpoint. Import-population keeps the Load List disposable and the Model layer stable: you can rebuild the Load freely, and the model changes only when you run the import.

3. Load List — Required Properties

Every Load List implementing this pattern carries the properties below, listed in dependency order (each may depend on earlier ones).

Property Type Input / Formula Purpose

Code Text Manual input / import Source system identifier
Parent_Code Text Manual input / import Source code of the parent node; BLANK for root nodes
Parent_Internal_ID Dimension (self) Formula Resolves Parent_Code to the actual dimension item in this list

Formula: Parent_Internal_ID

// Resolves the parent code to the actual parent item in this list.
// Returns BLANK for root nodes (no parent).

Why self-referential? A dimension property can reference the list it belongs to. This enables chained property access (x.Parent_Internal_ID.Parent_Internal_ID), which is the foundation of the entire ancestor walkout pattern.

How it works: IFDEFINED evaluates to its second argument when the first argument resolves to a defined (non-BLANK) value. Each chained hop peels back one generation. The deepest test (most hops) is checked first; the first match wins.

Node type Result
Root — no parent (L1) 0
One level down (L2) 1
Two levels down (L3) 2
… …
Six levels down (L7) — deepest supported 6
Seven or more levels down 6 (capped — see Depth Exceeded)

Built for 7 levels from day one — spare capacity sits parked, unused, until ever needed. The Level 1–Level 7 resolvers and the Cost Centre L1–L7 slots are all written and working immediately, whatever depth today's data actually reaches. A model whose hierarchy only goes 2 or 3 deep simply leaves the deeper levels dormant: the L4–L7 slots resolve to blank or promote harmlessly, the resolver formulas cost nothing, and nothing needs building. The 7-level ceiling is free headroom, not future work — if the hierarchy ever deepens, activation is create-one-list plus flip-one-property (Section 7), because the logic underneath is already there and tested. You never build levels you don't need, and you never have to retro-fit the ones you do.

3.3 Depth Exceeded (Warning Property)

Property Type Purpose
Depth Exceeded Text Fires a warning if any node sits deeper than the supported ceiling

IMPORTANT — do NOT use IF(Depth > 6, ...). The Depth property is itself capped at 6 by its IFDEFINED chain, so Depth > 6 is unreachable dead code: an item 8 levels deep still reports Depth = 6 and the warning never fires. Overflow must be detected directly, by testing whether a 7th parent hop resolves.

Formula: Depth Exceeded

// Fires when a 7th parent hop resolves, i.e. the item sits deeper than the
// 7-level ceiling. Cannot be derived from Depth, which is capped at 6.
// To extend the ceiling to 8 levels: add one more Parent_Internal_ID hop here.
IFDEFINED('Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
  "⚠️ Depth > 6 — hierarchy exceeds 7 levels. Extend Depth, Level properties, and parent hops (Section 6).",
  BLANK)

Important: Monitor this property after every data load. A non-blank value means source data has grown beyond the template ceiling and extension is required before those nodes will aggregate correctly.

Tuning the Ceiling to Your Data (recommended)

Two distinct numbers, easily conflated. The frame is the height the hierarchy actually has — a property of the tree. The ceiling is the depth the resolvers support — a property of this template. Tuning means setting the ceiling to the frame. See Concepts §1.

The formula above sets the ceiling at the template maximum of 7 levels. In practice, a stronger approach is to tune the ceiling to the actual known depth of your hierarchy — if the data is meant to be 3 levels deep, set Depth Exceeded to fire at Depth > 2. This turns the warning from a distant overflow catch into an active data-integrity assertion: it fires the instant any item appears deeper than the hierarchy is designed to be, catching a bad parent link or an unexpected source row immediately rather than letting it resolve silently through levels that should not exist.

// Depth Exceeded — tuned ceiling: 3 levels (Depth 0–2 = Element Level 1–3).
// Fires the moment any item sits at Depth > 2, signalling a data error or an
// intended deepening. All Level calculations at L4+ are blank by design while this holds.
// To raise the ceiling: change the threshold from 2 to the new maximum Depth.
IF('Cost Centre Load'.Depth > 2,
    "⚠️ Depth > 2 — hierarchy exceeds 3 levels. Item is beyond the known ceiling.",
    BLANK)

Paired with this, the unused deeper resolvers (Level 4–7) are commented out to a single BLANK rather than left computing — the dimension only calculates the levels it actually has, and the full reactivation recipe lives in the comment so growth is a controlled, documented step:

// Level 4 ancestor resolver — INACTIVE (tuned ceiling: 3 levels, Depth 0–2).
// Depth Exceeded fires for any item at Depth > 2; L4+ returns BLANK by design.
//
// TO REACTIVATE when the hierarchy grows beyond 3 levels:
//   1. Raise the 'Depth Exceeded' threshold: Depth > 2  →  Depth > .
//   2. Delete the BLANK line below and uncomment the formula block.
//
// Depth 0 (root)          : BLANK — root sits above L4
// Depth 1–3 (short branch): itself — leaf promotion, acts as its own L4 grouping
// Depth 4 (L5)            : Parent (1 hop)   Depth 5 : 2 hops   Depth 6 : 3 hops
// IF('Cost Centre Load'.Depth == 0, BLANK,
// IF('Cost Centre Load'.Depth <= 3, ITEM('Cost Centre Load'.Name, 'Cost Centre Load'.Name),
// IF('Cost Centre Load'.Depth == 4, 'Cost Centre Load'.Parent_Internal_ID,
// IF('Cost Centre Load'.Depth == 5, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID,
// IF('Cost Centre Load'.Depth == 6, 'Cost Centre Load'.Parent_Internal_ID.Parent_Internal_ID.Parent_Internal_ID,
// BLANK )))))
BLANK

Design assumption. The tuned ceiling treats a fired Depth Exceeded as a stop-and-fix signal: deepening is an exceptional, reviewed event, not routine. When real data legitimately deepens, the warning fires and the deeper items blank at L4+ until someone runs the reactivation recipe — so the design relies on a human seeing and acting on the warning, rather than the structure absorbing growth silently. For a governed hierarchy (cost centres, entities) this is the right trade: a loud warning plus a controlled extension beats silent auto-absorption. If you instead want growth absorbed automatically without intervention, keep all seven resolvers live (provisioned but dormant, per Section 3.5) rather than tuning the ceiling down.

3.4 Level Properties (Ancestor Walkout)

Level properties answer the question: "Who is my ancestor at Level N?" There is one property per supported level: Level 1 through Level 7. These are internal resolvers; they stay named Level N because the number genuinely denotes an absolute depth position.

The Leaf Promotion Rule

In a ragged hierarchy, shallow nodes have no ancestor at deeper levels. Rather than returning BLANK (which would drop them from grouped reports and aggregations), they promote themselves: a node sitting at L3 is its own Level 3, Level 4, Level 5… all the way down.

This is the mechanism that makes the pattern loss-proof: it guarantees every node has a defined value at every level, so no blank ever reaches a BY, a fetch, or a filter.

Worked example — Warehouse from the Section 1 tree (Depth 2, sits at L3):

Level 1 → Company     (2 parent hops)
Level 2 → Operations  (1 parent hop)
Level 3 → itself      (Depth equals target — at-level)
Level 4 → itself      (leaf promotion)
Level 5 → itself      (leaf promotion)
Level 6 → itself      (leaf promotion)
Level 7 → itself      (leaf promotion)

Formula Pattern: Level N

The target depth for Level N is N−1. For a node at Depth D, its Level N ancestor is D − (N−1) parent hops away:

What is N? It is the level number you are building (1–7), not a Pigment variable. This block is a template: pick the property (e.g. Level 2 → N = 2), substitute the number, and the branches resolve to hard literals. N never appears in the finished formula — that is why the concrete formulas below contain only Depth == 0, Depth == 1, … and no N. Likewise the first line’s and N > 1 is a note to the reader (“this BLANK line only exists in Level 2 and above”), not something you type — in Level 1 it is absent; in Level 2+ it becomes a plain IF(Depth == 0, BLANK, …).

3.5 Curated Dimension Properties (L1–L3 Active, L4–L7 Stubs)

These properties join the Load List to the standalone curated dimension lists. They exist in two states:

Active (L1–L3)

  • Type: Dimension (references the curated list)
  • Formula: ITEM() matches the Level N ancestor name to the curated list
// Cost Centre L1 — Active dimension join.
// Maps the Level 1 ancestor name to the Cost Centre L1 dimension list.
ITEM('Cost Centre Load'.'Level 1'.'Name', 'Cost Centre L1'.'Name')

Stub (L4–L7)

  • Type: Text (no curated list exists yet)
  • Formula: Returns the ancestor name as plain text — BLANK when no ancestor exists
// ─────────────────────────────────────────────────────────────
// Cost Centre L4 — Template stub (Text)
// ─────────────────────────────────────────────────────────────
// STATUS: Stub — returns the Level 4 ancestor name as text.
// BLANK for all current data (max Depth 2).
//
// WHEN TO ACTIVATE:
// Source data contains items at Depth 3 or deeper.
//
// UPGRADE STEPS:
// 1. Create dimension list "Cost Centre L4" with a Name property.
// 2. Populate it by import from the distinct values in this property.
// 3. Change this property type: Text → Dimension (Cost Centre L4).
// 4. Replace this formula with:
//    ITEM('Cost Centre Load'.'Level 4'.'Name', 'Cost Centre L4'.'Name')
//
// DEPENDENCY: Level 4 property must remain on this list.
// CEILING: Depth formula and Level properties support up to Depth 6 (7 levels).
// ─────────────────────────────────────────────────────────────
'Cost Centre Load'.'Level 4'.'Name'

The instruction block is embedded directly in the formula comment so the upgrade path is self-documenting — no external reference required.

Hide the stub properties until they are needed. The Cost Centre L4–L7 stubs exist so the resolver machinery is complete to 7 levels, but in a hierarchy only 2–3 deep they carry no useful value and would clutter the property list. Set them to hidden on the Load List: the property, its formula, and — importantly — its embedded activation instructions all remain intact and travel with the model, but they stay out of sight until a future deepening calls for them.

This pairs with the comment convention: the activation steps live in the stub’s own formula comment (shown above), not in an external document. So when the day comes to activate Cost Centre L4, you unhide the property and the exact upgrade procedure is already there in front of you, in the formula editor, at the precise place the change is made. Documentation that travels with the property cannot go stale or get lost the way an external note can. Hidden-but-documented is the goal: zero clutter today, zero archaeology tomorrow.

3.6 Node Classification — Child Count & Boolean Flags (optional QA layer)

The Level resolvers and Depth Exceeded are the working engine of the pattern — they resolve the hierarchy and catch overflow. On a staging Load List it is also useful to carry a small set of human-readable classification flags — is this item a root, an intermediate rollup node, or a leaf — so a reviewer can eyeball how each row was classified before it is promoted to the curated dimensions. These flags are a QA/validation layer, not a building block: the Level formulas do not depend on them. Built correctly, they surface real structural information the Level properties alone do not make visible.

Why leaf detection needs a downward signal

The instinct is to classify nodes from their ancestors: “a root has no parent, an intermediate has a parent but no grandparent, a leaf has both.” This upward logic is wrong for ragged hierarchies, and the failure is specific. Consider a parent whose children are a mix — one is a real rollup node with its own children, the others are bare leaves:

Marketing (root)
├── Direct Leaf A   → parent ✓, grandparent ✗
├── Direct Leaf B   → parent ✓, grandparent ✗
└── Sub-Rollup      → parent ✓, grandparent ✗   ← has children
    ├── Leaf C      → parent ✓, grandparent ✓
    └── Leaf D      → parent ✓, grandparent ✓

Direct Leaf A and Sub-Rollup are siblings — same parent, same depth, identical ancestors. Any test that looks upward gives them the same answer, so ancestor-blank logic classifies the direct leaves as rollup nodes: wrong. The distinguishing fact — does this item have children — lives downward, and upward logic is structurally blind to it. Sibling nodes are the case that breaks every upward-only classifier.

The rule. Position facts (root, depth, level) can be read upward — an item’s ancestors genuinely tell you where it sits. Type facts (is this a leaf) cannot — “leaf” means “nothing points at me as its parent,” which is a downward fact. Leaf detection therefore requires a reverse child-count, never an ancestor test.

Child Count — the downward signal (BY COUNT must be a metric)

The count of an item’s children is a reverse traversal: group all items by their Parent_Internal_ID and tally how many land on each parent. A platform note worth knowing: in Pigment this BY COUNT reverse-aggregation cannot be written as a List Property — it must be built as a metric dimensioned by the Load List, which is then referenced back by the classification properties. That constraint applies to BY COUNT specifically, not to leaf detection generally: the MATCH idiom (Section 2) answers the same downward question as a plain list property, at the cost of an untested profile on very large lists.

// Cost Centre Child Count — METRIC (not a list property).
// Counts items whose Parent_Internal_ID points to this item.
// BLANK/zero for leaves (never named as anyone's parent); > 0 for real nodes.
'Cost Centre Load'.Internal_ID [BY COUNT: 'Cost Centre Load'.Parent_Internal_ID]

The classification flags

With the child-count metric available, the three flags classify every item into exactly one of root / rollup / leaf. Note that CC Rollup and CC Base are sourced from the child count (downward), not from ancestor-blanks — this is what makes them correct for the sibling case. Both also gate on Depth Exceeded being blank, so an over-deep item falls out of all three for review rather than misclassifying.

// CC Parent — TRUE for root items (no parent).
IF(ISBLANK('Cost Centre Load'.Parent_Internal_ID), TRUE, BLANK)
// CC Rollup — TRUE for intermediate nodes: has a parent, HAS children, within ceiling.
IF(ISDEFINED('Cost Centre Load'.Parent_Internal_ID)
   AND NOT ISBLANK('Cost Centre Child Count')
   AND ISBLANK('Cost Centre Load'.'Depth Exceeded'), TRUE, BLANK)
// CC Base — TRUE for leaf nodes: has a parent, NO children, within ceiling.
IF(ISDEFINED('Cost Centre Load'.Parent_Internal_ID)
   AND ISBLANK('Cost Centre Child Count')
   AND ISBLANK('Cost Centre Load'.'Depth Exceeded'), TRUE, BLANK)

The three are mutually exclusive and complete. A childless root resolves to CC Parent only — the ISDEFINED(Parent) guard on CC Rollup and CC Base keeps a root out of both, so it is never mistaken for a leaf. Verify on real rows: a known rollup node should read CC Rollup, a known bare leaf should read CC Base, including for mixed siblings under the same parent.

Use these as an audit layer, not an engine. The flags are a staging-file QA surface — a quick visual check that each item classified as expected before promotion to the curated dimensions. They read correctly because they are fed by the child count; the identical-looking flags fed by ancestor-blanks are the flawed version and must not be used. Keep the Level resolvers as the engine underneath; let the flags sit alongside as the human-readable check.

The Load List computes the hierarchy; the curated lists carry it into production. Each curated list needs two things: the right items, and a parent-link property that lets views walk up the hierarchy. Both are populated by import from the Load List (never a subset or live-derived link — Section 2).

4.1 Structure — One Parent Link per List

Each curated list carries a dimension property referencing the list one level above it. Immediate parent only — higher ancestors are reached by chaining.

List Properties Parent link
Cost Centre L1 Name (unique) — (top of hierarchy)
Cost Centre L2 Name (unique), Cost Centre L1 Dimension → Cost Centre L1
Cost Centre Name (unique), Cost Centre L2 Dimension → Cost Centre L2

Walking up the hierarchy from the leaf list is then chained property access:

'Cost Centre'.'Cost Centre L2'                    // immediate parent (L2)
'Cost Centre'.'Cost Centre L2'.'Cost Centre L1'   // two levels up (L1)

Optional flattening — putting a higher level directly on the leaf.

The leaf list only holds a link to the list one level above it (Cost Centre L2). It has no direct L1 property — to reach a leaf’s BU you must chain two hops: 'Cost Centre'.'Cost Centre L2'.'Cost Centre L1'. That chain works fine inside a formula.

The problem: some parts of Pigment want a single property on the leaf, not a chain — board selectors, view groupings, and page dimensions let you pick a property of the dimension but won’t accept a two-hop expression. So “group this board by BU” or “add a BU selector” has nothing to point at, because L1 sits two hops away.

The fix: add one derived property directly on the leaf — call it Cost Centre L1 — whose formula is the chain:

// Flattened L1 on the leaf — collapses the two-hop chain into
// a single directly-selectable property.
'Cost Centre'.'Cost Centre L2'.'Cost Centre L1'

Now L1 is a first-class property of the leaf, alongside L2, and any selector, grouping, or page dimension can point straight at it. It is no extra import because it builds no list and loads no data — it is a pure formula reading properties that already exist, recomputing itself at no load-time cost.

When to bother: only if something actually needs a higher level as a leaf property (a BU selector, a BU-level grouping on a leaf-dimensioned board). If everything that needs L1 lives inside formulas — where the chain already works — you don’t need it. The same one-line trick flattens any level (L1, L2, deeper) onto the leaf if heavy board work calls for it.

How Many Flat Level Properties Should the Leaf Carry?

With a hierarchy up to 7 levels deep, a natural question is whether to add all seven levels as flat properties on the leaf dimension. The answer is no — you add a flat level property only where something consumes that level as a leaf property, and you provision them one at a time as concrete needs arise. Reflexively flattening all seven is clutter, not completeness.

The reason is that there are three different things that might make you want a level available, and only one of them needs a flat property:

What you want to doWhat it needsFlat property required?
Roll up in views — subtotals by BU, department, etc.Nothing extra. Group the view by Cost Centre L2 and the view walks the parent-link chain (L2 → L1) itself.No — the chain already does this.
Filter / isolate leaves by ancestor in a formula (Section 8.4)The chain, referenced in the formula: 'Cost Centre'.'Cost Centre L2' = X.No — available without adding anything.
A selector, grouping, or page dimension that must point at one specific levelA flat property on the leaf for that level only (the flattening trick above).Yes — but only for the level that specific control uses.

So across a full seven-level hierarchy the realistic count of flat level properties is small — often just one or two (commonly L1/BU, because BU selectors and BU-grouped boards are common), and rarely all seven. Every flat property you add that nothing points at is dead weight: one more attribute on the dimension to read past, maintain, and later puzzle over (“why is L5 flattened — what uses it?”). Unused derived properties are cheap to compute but not free to understand.

Watch the naming — structural vs convenience properties. The leaf already carries Cost Centre L2 as its genuine immediate-parent link — a structural property the walkout and imports depend on. A flattened Cost Centre L1 is a different thing: a derived convenience for the UI, not a structural link. If you flatten several levels, make the distinction visible — for example prefix the convenience ones (Sel_L1, Flat_L1) — so a later reader never mistakes a flattened lookup for a real parent link and builds on it wrongly.

Rule of thumb: the chain is the default; a flat level property is an exception you justify by naming a concrete UI consumer for it. Provision the exceptions as they appear — not all seven up front.

4.2 Import Source — Distinct (Level N, Level N−1) Pairs

Each curated list is populated by an import whose source is a view of the Load List, showing the distinct Level N names alongside their Level N−1 names. The second column fills the parent link.

Target list Source columns (from Load List) Mapping
Cost Centre L1 distinct Level 1.Name → Name (unique key)
Cost Centre L2 distinct Level 2.Name + Level 1.Name → Name (key), → Cost Centre L1 (parent)
Cost Centre distinct Level 3.Name + Level 2.Name → Name (key), → Cost Centre L2 (parent)

The dimension-typed parent property is resolved on import by matching the incoming text against the parent list's Name — the standard Pigment import behaviour for dimension properties.

The (Level N, Level N−1) pair is functionally dependent: every Load List row sharing a Level N ancestor also shares that ancestor's Level N−1, so the distinct extract yields exactly one unambiguous parent per item. No deduplication conflicts are possible.

4.3 Run Order & Reload Behaviour

  1. Import top-down: L1, then L2, then the leaf. Each import needs its parent list already populated, or the parent-link property cannot resolve.
  2. Same order on every reload. Imports are upserts keyed on Name, so re-running is safe and idempotent.
  3. The import is the gate. Refresh and validate the Load List first (Section 10), then run the imports to update the model. Models and apps only change when you choose to run the import — the Load List can be rebuilt freely in between without touching production.
  4. Leaf promotion flows through cleanly. A promoted shallow leaf lands in every list at and below its own level, with parent links chaining through its own promoted entries. Example — Finance (Depth 1): appears in L2 and the leaf list; the chain resolves leaf Finance → L2 Finance → L1 Company. Rollup stays correct with no special-casing.

Why not a List Subset or a formula-derived link? Both would make the curated lists a live reflection of the Load List, removing the validation gate: a bad load, a mid-load state, or a source-format change would reach models instantly. Import-population is deliberate and checkable; that is the whole point of the two-tier separation. This rule is non-negotiable in this pattern.

6. Extension Guide — Adding Level 8

If source data exceeds 7 levels (Depth Exceeded fires), extend the template as follows. Every change is mechanical — copy the pattern, increment the number.

  1. Update Depth. Add one more IFDEFINED branch at the top, testing 7 chained parent hops and returning 7.
  2. Update Depth Exceeded. Extend the overflow test from 7 chained hops to 8, and update the warning text.
  3. Update Level 1–7 formulas. Add one new branch to each: IF(Depth == 7, <7−(N−1) parent hops>, ...) immediately before the safety fallback.
  4. Create Level 8 property. Follow the standard Level N pattern with target Depth = 7.
  5. Create the new intermediate curated list and its parent-link import (see Section 7).

Rule of thumb: every +1 level touches 11 properties — Depth, Depth Exceeded, the seven existing Level formulas, the new Level 8, and the new curated stub. Purely mechanical.

7. Activating a Stub & Deepening the Hierarchy

7.1 Activating a Stub Dimension

When source data begins to include items at Level 4:

  1. Identify distinct L4 values — filter the Load List to rows where Cost Centre L4 is not blank.
  2. Create the curated dimension list Cost Centre L4 with a unique Name property and a parent-link dimension property referencing Cost Centre L3's list. The parent link is not optional — it carries the rollup.
  3. Populate by import from the distinct values in Step 1.
  4. Update the Load List property Cost Centre L4: Text → Dimension.
  5. Update the formula to ITEM('Cost Centre Load'.'Level 4'.'Name', 'Cost Centre L4'.'Name').
  6. Verify — spot-check items; items at Depth ≤ 2 resolve to themselves via leaf promotion, which is correct.

7.2 What Happens to the Leaf on Deepening

Because the metric-facing dimension is named Cost Centre (not Cost Centre L3), deepening does not rename or restructure it. When a former leaf gains children:

  • Under promotion, the former leaf's deeper Level properties already return itself, so the leaf import (now sourced from the deeper Level column) adds the new children and keeps the former leaf as a postable parent-with-own-value. Existing members and their stored data survive — membership is additive, not a rebuild.
  • A new intermediate curated list is created for the newly-real middle tier, built exactly like L2 was.
  • Rules and metrics do not change, because they reference 'Cost Centre' and the stable upper levels — never a leaf-position number.

7.3 The Naming Rule

The metric-facing dimension carries a position-free name — Cost Centre, never Cost Centre L3. Numbered names are stable only for positions that don't move, and deepening moves the leaf. The upper levels keep top-anchored numbers (Cost Centre L1, L2, …) because roots are always L1 and deepening only adds tiers below. Result: deepening costs two actions — create the new intermediate list, switch the leaf import's source column from Level N to Level N+1 — and zero rule changes.

8. Calculation Patterns — Property-Reference Rules

Most business rules need to combine a value stored at the leaf with a value stored higher up the hierarchy. A simple example: “apply each department’s overhead rate to every cost centre in it” — the spend sits at the leaf (each cost centre), but the rate sits at the department (an L2 level).

There are two ways to build a rule like that, and they differ in what dimension the calculation itself runs on:

  1. Dimension the calculation by the level list. The rule’s output metric is built on Cost Centre L2, so it computes one value per department. Simple, but it forces the result to L2 grain, and it only sees leaves that have an L2 — the failure mode this whole document is about.
  2. Keep the calculation at leaf grain, and reach up for the higher value. The rule’s output metric stays on the leaf dimension Cost Centre (one value per cost centre), and where it needs the department rate it fetches it by following the leaf’s parent-link property up to L2. The hierarchy is used as a lookup path, not as the shape of the result.

This pattern prefers the second. The rule lives at leaf grain — the one place every cost centre exists — and pulls higher-level values down as needed. The only thing that stays dimensioned by a level is a value people actually type in at that level (a department rate someone enters once per department); the calculation that consumes it stays at the leaf. Section 8.3 lays out exactly which of your blocks sit at a level and which sit at the leaf.

8.1 The Pattern

Rules are dimensioned by the leaf list (Cost Centre). Where a rule needs a value held at a higher level — a rate, a driver, an allocation basis — it fetches it through the chained parent-link properties rather than by dimensioning the rule itself at that level:

// Apply a BU-level rate to every leaf cost centre.
// 'Rate by L2' is dimensioned by Cost Centre L2 (input grain).
// The calculation stays at leaf grain; the mapping fetches each
// leaf's parent rate through the parent-link chain.
'Revenue by Cost Centre' * 'Rate by L2'[BY: 'Cost Centre'.'Cost Centre L2']

Because promotion guarantees Cost Centre L2 is never blank, this BY resolves for every leaf — including ragged ones, which land on their own promoted L2 entry. No silent drop.

8.2 Two Prerequisites

Promotion underneath. Property-reference rules are safe only because promotion guarantees the properties are never blank. A blank level fails a lookup by the same mechanism it fails a BY — there is no member to resolve to — so the loss-proofing and the rule pattern are the same guarantee viewed twice.

Reference the curated leaf list, never the Load List. The Load List is staging (Section 2). Keying business rules to 'Cost Centre Load' references breaks the tier separation — every re-import's item churn would ripple into calculation logic. The curated leaf list already carries the parent chain (Section 4), so rules reference 'Cost Centre'.'Cost Centre L2' and the staging layer stays disposable. Load and Model are separate concerns: imports may rebuild the Load List freely; rules bind only to the Model layer.

Note the distinction: parent relationships are derived on the Load List (at import time, via the walkout properties), but production formulas bind to the curated lists, where the import has materialised the result. Derivation location and binding target are different questions — conflating them is how staging references leak into production logic.

8.3 What Level Dimensions Remain For

Level lists don't disappear — they stop being calculation dimensions and persist for two roles:

Use case Structure
Input grains — planners enter rates, targets, assumptions at a level Metric dimensioned by that level list (e.g. Rate by L2). An input "per BU" needs an L2-shaped container; it cannot be typed into a leaf-grain block without duplication.
Calculation — rules combining data across grains Leaf-dimensioned metric; higher-level values fetched via the parent-link chain (8.1)
Reporting — subtotals, rollups, level views View grouping over leaf metrics, or level-dimensioned metrics fed by BY — both safe, because promotion leaves no blanks

8.4 Pulling an Assumption Down From a Parent

This is the everyday job of a planning model: an assumption is entered against a parent node — an overhead rate, a headcount driver, a growth factor — and every leaf underneath needs to consume it. How you fetch it depends on whether you know which level the assumption sits at. There are three situations, in increasing generality.

Case A — the assumption is held at one known level

If the rate is entered per department (an L2 metric, Rate by L2), every leaf reaches it in a single hop, because the leaf already carries its L2 ancestor as a property:

// Each leaf pulls its own department's rate. One hop, no climbing.
'Spend by Cost Centre' * 'Rate by L2'[BY: 'Cost Centre'.'Cost Centre L2']

Promotion guarantees Cost Centre L2 is never blank, so this resolves for every leaf including ragged ones (which land on their own promoted L2 entry). This is the common case and needs nothing clever.

Case B — the assumption may sit at different levels, so climb to the nearest ancestor that has one

Often an assumption is entered at whatever level makes business sense, and filled in sparsely: a rate is set for some departments (L2), but where a department has none the leaf should inherit its BU’s rate (L1), and failing that a global default. You don’t know per-leaf which level will supply the value — so you check from the most specific level upward and take the first one that has a value. An IFBLANK cascade does exactly this, and is the climb:

// Nearest-ancestor inheritance: try the leaf's own level, then L3, then
// L2, then L1 — stop at the first ancestor that actually carries a rate.
// The cascade discovers which level supplies the value; you don't specify it.
IFBLANK( 'Rate'[BY: 'Cost Centre'.'Cost Centre L3'],
IFBLANK( 'Rate'[BY: 'Cost Centre'.'Cost Centre L2'],
IFBLANK( 'Rate'[BY: 'Cost Centre'.'Cost Centre L1'],
         'Global Default Rate' )))

Read it top to bottom: “use the L3 rate if one is set; otherwise the L2 rate; otherwise the BU rate; otherwise the global default.” Because each level property is populated for every leaf (promotion), the only thing that can be blank is the rate table itself at that level — which is precisely the “no value entered here” condition you want the climb to skip past. Order the cascade most-specific-first so the most local override wins, which is almost always the intended precedence (a department rate should beat the BU default).

Case C — you are given an element and must find whether it is above a given leaf

Sometimes the assumption arrives keyed to a specific element — “this rate applies to everything under node X” — and X could be at any level. You need each leaf to answer “is X one of my ancestors?” without knowing X’s level. Because the leaf carries all its ancestors as properties, this is a flat scan across the ancestor slots — X can only match the slot at its own level:

// Does element X sit anywhere above this leaf? Scan all ancestor slots.
// No need to know X's level — it matches exactly one slot if it's an ancestor.
'Cost Centre'.'Cost Centre L1' = X
  OR 'Cost Centre'.'Cost Centre L2' = X
  OR 'Cost Centre'.'Cost Centre L3' = X
  // ... up to the leaf itself

Wrap that as a boolean helper property on the leaf (“Is under X”) and any rule can reuse it to apply X’s assumption to exactly the leaves beneath it. This is the general form of “climb the tree looking for a node”, but it is not an iterative climb — the flattened ancestor properties turn it into a single flat comparison, which is why the balanced structure makes it cheap.

The through-line: in every case you are reading the leaf’s own materialised ancestor properties — one hop when you know the level (A), an IFBLANK cascade when you must find the nearest filled level (B), a flat slot-scan when you must locate a named element (C). None of them traverses the tree at query time, because the walkout already did that once at the dimension layer. That is what the pre-computed levels buy you.

8.5 Isolating Leaves by Ancestor

Because every level is a property of the leaf dimension, isolating leaves by any ancestor at any level is a flat property comparison — no tree traversal:

[FILTER: 'Cost Centre'.'Cost Centre L1' = "G&A"]          // by BU
[FILTER: 'Cost Centre'.'Cost Centre L2' = "Finance Team"] // by department

All descendants of an arbitrary node, without knowing the node's level, is a disjunction across the ancestor slots:

[FILTER: 'Cost Centre'.'Cost Centre L1' = P
      OR 'Cost Centre'.'Cost Centre L2' = P
      OR 'Cost Centre'.'Cost Centre L3' = P ... ]

This is exact precisely because no slot is ever blank: a non-descendant can never equal P at any slot, and complement queries (<> P) are well-defined because there is no blank to fall into undefined territory.

8.6 Why This Wins

A single calculation grain eliminates the grain-mismatch errors that multiply when rules are scattered across level-dimensioned blocks. Hierarchy restructures — a BU split, a re-parenting — flow through the imports and parent links without touching rule logic, because no rule names a level structure; each rule names a path. And because promotion leaves no blanks, every level is safe for every calculation type, with no usage constraint to police.

9. Design Principles & Rationale

Why not use a recursive formula directly on metrics?

Pigment formulas are not recursive: they cannot traverse a variable number of parent hops at query time. The Level properties pre-compute every ancestor relationship once, at the dimension layer, making metric formulas simple lookups.

Why leaf promotion instead of BLANK?

If shallow leaves returned BLANK at deeper levels, they would aggregate into an "unknown" bucket or vanish from grouped reports and aggregations entirely — a silent data-loss failure that typically surfaces only when someone notices a total doesn't reconcile. Leaf promotion guarantees every node participates in every level's subtotal, preserving complete aggregation trees regardless of branch depth.

The failure is real and confirmed. A blank level property is excluded from BY aggregation with no error and no null bucket:

Empirically confirmed (July 2026): a four-item fixture (two items with a populated intermediate property, two with it blank, values totalling 28) aggregated via [BY: ...blank-bearing property] returned 13 — the two blank items vanished silently. The control aggregation to a fully-populated property returned 28, and a view grouped over the same blanks also totalled 28. The formula and view layers disagree by exactly the blank-mapped rows.

Promotion removes the blank, so this failure cannot occur anywhere in the model. That is the entire reason to pay the pattern's modest cost.

There is also a quieter simplicity advantage. At any given depth, an item may be a real internal node (it has children) or a ragged leaf (it does not) — and at that item's own level the two must resolve differently: the real node is its own ancestor there, the leaf has no ancestor there. Depth alone cannot tell them apart, so a blank-based resolver has to compute an extra Has_Children flag (a reverse child-count) and gate every level on it. Promotion never has to make this distinction: “at your own depth, return yourself” is correct for both — a real node returns itself legitimately, and a leaf returning itself is exactly the intended promotion. So the balanced pattern is not only loss-proof but structurally simpler at this ambiguous spot: no Has_Children flag, no per-level gate, one uniform rule.

Why dimension metrics by the leaf list only?

Dimensioning a metric by L1 × L2 × L3 creates a sparse cross-product in which most combinations are invalid ancestry paths — dead space, with nothing preventing data landing in an invalid cell. Dimensioning by the leaf list alone makes every cell valid by construction: the L2 and L1 rollups derive from the parent-link properties, so the hierarchy is displayed in views rather than baked into the metric's shape. Activating a deeper level then never requires restructuring existing metrics — only the leaf reference changes.

Why keep the Load List separate, and import-populated?

  • Source isolation: raw import data (codes, parent codes, raw names) is isolated from the clean production dimensions used in metrics.
  • The validation gate: because the curated lists are import-populated rather than subsets or live-derived, loads can be refreshed and checked in the Load List before they reach models. Nothing propagates to calculations until the import runs.
  • Reload safety: re-importing the Load List does not disturb the curated lists or any metrics dimensioned by them.
  • Auditability: the Load List shows every node, including intermediate nodes that may not be meaningful planning dimensions in their own right.

Why embed upgrade instructions in formula comments?

Formula comments are the only documentation that travels with the model and is guaranteed to be in front of a modeller when they open the property. External documents go stale; comments in the formula editor surface exactly where the decision is made.

Why Text stubs instead of leaving properties out entirely?

A Text stub immediately shows what value would resolve if the dimension existed; it makes data quality visible (L4 values appearing means activation is needed); and the upgrade is a type change plus a one-line formula swap — no structural rework, reversible.

10. Validation Checklist

After loading or reloading source data — and before running the imports that update the model — verify:

  • [ ] Depth Exceeded is entirely blank (no depth overflow)
  • [ ] All root nodes have Depth = 0 and Level 1 = themselves
  • [ ] All leaf nodes return themselves for every Level N deeper than their own depth (leaf promotion — no blanks below own depth)
  • [ ] Cost Centre L1–L3 resolve to non-blank Dimension values for every node at or below that level
  • [ ] Cost Centre L4–L7 stubs show non-blank text only for items at Depth ≥ 3 / 4 / 5 / 6 respectively
  • [ ] Node count reconciles: Load List item count equals the source extract row count — no silent exclusions
  • [ ] Curated counts reconcile: each curated list's item count equals the count of distinct Level N values in the Load List
  • [ ] Every item in Cost Centre L2 and Cost Centre has a non-blank parent link (a blank means the import ran out of order or a name failed to match)
  • [ ] Spot-check one deep node: verify its full ancestor chain against source data

Only once these pass are the imports run to update the model — the separation gate in practice.
