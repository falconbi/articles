IF() Dimensional Scope Contamination

A 13-branch IF formula silently inflated reconciliation totals by exactly 12× — no error, no warning, no failed validation. A real-world case study and the isolation pattern that resolves it permanently.

Pigment IF scope · Pattern Cost apportionment model July 2026
Start here — the plain-language version

The simple version

Imagine you're asked to calculate the monthly salary cost for your Finance team. Simple enough. Now imagine that somewhere else in the same spreadsheet, a formula for a completely different team references a list of 12 project codes.

Without you realising it, the tool silently applies those 12 project codes to your Finance salary calculation too — multiplying it by 12. Your $100K salary cost suddenly shows as $1.2M. The formula looks correct. No error message appears. The number is just wrong.

That is exactly what happened here.

Why it happens

Most planning tools — and the AI that helps build them — are designed to write formulas the same way Excel or SQL does: one big formula with many conditions, each handling a different scenario. The problem is that Pigment works differently under the hood. When a formula has multiple conditions, the tool looks at all of them at once before deciding which one to use. If any one condition involves a list of 12 items, the tool temporarily applies that list of 12 to every other condition in the formula — even the ones that have nothing to do with it.

The end result: numbers that are a clean multiple of the list size. In this case, 12× too high.

Why it's hard to catch

  • The formula passes all built-in checks — no errors, no warnings
  • The number looks plausible at a glance (it's still a large, round-ish figure)
  • It only becomes obvious when you know what the answer should be and notice the exact multiple

How it was fixed

Rather than one large formula handling everything, the logic was split into smaller, self-contained calculations — each handling only what it needs to, with no risk of unintended cross-contamination. Those smaller calculations are then combined into the final result.

Think of it like separating your Finance and Projects cost centres into their own tabs before combining them on a summary sheet — rather than trying to do everything in one cell.

This is not a one-off bug. It is a known characteristic of how Pigment processes formulas — one that affects human modellers and AI alike. Complex logic with many conditions should always be broken into smaller, named calculations. Each piece is easier to check, easier to explain, and immune to this type of silent contamination.

01 — Context

Context

This document describes a real bug encountered while building a cost apportionment model in Pigment. A single reconciliation-control metric silently returned values inflated by an exact factor of 12, with no error and no failed validation. The root cause is the interaction between Pigment's eager evaluation of IF, its automatic dimension alignment, and its default aggregation on assignment.

The purpose is threefold: to document the mechanism precisely enough to be reproducible, to establish exactly where Pigment's documentation does and does not cover it, and to record the isolation pattern that fixes it. Documentation references point to the current Pigment Knowledge Base (kb.pigment.com) and are collected in the References section.

02 — The model

The model

Target metric

CST_Ctrl_Input_Total ($)

  • Structure: [CST_ReconciliationCheck, CST_Version, Month]
  • Purpose: A reconciliation control metric. For each member of CST_ReconciliationCheck it returns a different calculated value — one per check type.
  • CST_ReconciliationCheck has 13 members: RC01, RC02, RC03, RC04, RC05, RC06, VAL00, VAL01, VAL02, VAL03, VAL04, VAL05, VAL06.

The formula

A single nested IF chain — one branch per ReconciliationCheck member. Every branch specifies both its True value and its False value (the next nested IF). This detail matters later: it is exactly the shape that suppresses the one dimension hint Pigment would otherwise surface.

IF(CST_ReconciliationCheck.Name = "RC01",
  'CST_Ctrl_Total_Overhead ($)',
IF(CST_ReconciliationCheck.Name = "RC02",
  'CST_Data_Pool_Settled_Amount ($)'[FILTER: CST_Activity.Name = "Input"][REMOVE: CST_CostPool, CST_Activity, CST_CostCenter],
IF(CST_ReconciliationCheck.Name = "RC03",
  'CST_Data_Pool_Settled_Amount ($)'[REMOVE: CST_CostPool, CST_Activity, CST_CostCenter],
IF(CST_ReconciliationCheck.Name = "RC04",
  IF(NOT ISBLANK('CST_A2A_Final_Settlement ($)'),
    'CST_Input_A2A_Base_Amount ($)'[SELECT: CST_ActivityDest.Name = "Input"][SELECT: CST_CostCenter.Name = "Input"]
  )[REMOVE: CST_Activity],
IF(CST_ReconciliationCheck.Name = "RC05",
  'CST_A2A_Final_Settlement ($)'[REMOVE: CST_Activity],
IF(CST_ReconciliationCheck.Name = "RC06",
  'CST_Ctrl_Total_Overhead ($)',
IF(CST_ReconciliationCheck.Name = "VAL00",
  IF(ISBLANK(CST_Input_Apportionment_Type), 1)[REMOVE SUM: CST_Account, CST_CostCenter],
IF(CST_ReconciliationCheck.Name = "VAL01",
  IF(CST_Input_Apportionment_Type = "Indirect",
    IF(ABS('CST_Input_Driver_Pct_Share (%)'[REMOVE SUM: CST_CostPool] - 100) > 0.1, 1)
  )[REMOVE SUM: CST_Account, CST_CostCenter],
IF(CST_ReconciliationCheck.Name = "VAL02",
  IF(CST_Input_P2P_Is_Active AND ISBLANK(CST_Input_P2P_Driver_Type), 1)[REMOVE SUM: CST_CostPool, CST_CostPoolDest],
IF(CST_ReconciliationCheck.Name = "VAL03",
  IF(CST_Calc_P2A_Driver_Value[REMOVE SUM: CST_Activity] > 0 AND ISBLANK(CST_Input_P2A_Driver_Type), 1)[REMOVE SUM: CST_CostPool],
IF(CST_ReconciliationCheck.Name = "VAL04",
  IF(CST_Input_A2A_Is_Active[SELECT ANY: CST_CostCenter.Name = "Input"] AND CST_Calc_A2A_Driver_Value[REMOVE SUM: CST_ActivityDest] = 0, 1)[REMOVE SUM: CST_Activity],
IF(CST_ReconciliationCheck.Name = "VAL05",
  IF(CST_Calc_A2SL_Driver_Value[REMOVE SUM: CST_ServiceLine] > 0 AND ISBLANK(CST_Input_A2SL_Driver_Type), 1)[REMOVE SUM: CST_Activity],
IF(CST_ReconciliationCheck.Name = "VAL06",
  0,
  BLANK
)))))))))))))
03 — The problem

The problem

Symptom

RC01 and RC06 returned ~$1.55B/month instead of the expected ~$129M/month.

The smoking number

$1.55B ÷ $129M = exactly 12. An exact integer ratio is the tell: the value has been multiplied by a dimension's member count.

The contaminating dimension

CST_ActivityDest — a dimension with 12 members.

Metrics implicated in the contamination

MetricReferenced byCarries CST_ActivityDest?
CST_Ctrl_Total_Overhead ($)RC01, RC06No
CST_Input_A2A_Base_Amount ($)RC04Yes
CST_Calc_A2A_Driver_ValueVAL04Yes

RC01 and RC06 return a metric that has no CST_ActivityDest. Two other branches — RC04 and VAL04 — reference metrics that do. That is the entire setup for the failure.

04 — What Pigment did

What Pigment actually did

Pigment's IF() uses eager evaluation: before selecting the winning branch, it evaluates the whole expression in the union of every dimension referenced anywhere across all branches. Because RC04 and VAL04 reference metrics dimensioned by CST_ActivityDest, that dimension enters the evaluation scope of the entire formula — including the RC01 and RC06 branches, which never mention it.

Step 1 — Scope determination

Pigment takes the union of all dimensions across all 13 branches. CST_ActivityDest (12 members) enters scope from RC04 and VAL04. All branches — including RC01 and RC06 — are now evaluated in a scope that includes CST_ActivityDest.

Step 2 — RC01 branch evaluated in the expanded scope

CST_Ctrl_Total_Overhead ($) has no CST_ActivityDest. To align it to the union scope, Pigment replicates its value ($129M) across all 12 ActivityDest members — the same behaviour as the [ADD Constant: …] modifier. RC01 now holds 12 cells, each containing $129M.

Step 3 — IF condition applied

ReconciliationCheck = RC01 → the RC01 branch wins. All 12 cells ($129M each) are selected.

Step 4 — Result assigned to the target metric

CST_Ctrl_Input_Total ($) has no CST_ActivityDest in its structure. Aligning the output back down uses the default [REMOVE SUM: …] behaviour. 12 × $129M = $1.55B.

Why no error fired

No error was raised. The formula passed all validation checks. The two halves of this mechanism — replication in Step 2 and summation in Step 4 — both appear on the Dimensionality Hints page as ordinary output-vs-structure alignment. What is nowhere documented is that an IF can chain them: replicate an unrelated branch across a foreign dimension, then sum it away, leaving a clean multiple.

05 — Failed fixes

Failed fix attempts

1
[BY CONSTANT: CST_ActivityDest] on the RC01/RC06 branches

Intended to pin ActivityDest at its context value.

✗ Failed — BY CONSTANT holds the dimension at its context value but does not remove it from scope. The 12 cells still existed and were still summed on assignment.
2
[SELECT: CST_ActivityDest.Name = "Input"] on CST_Ctrl_Total_Overhead ($)

SELECT collapses a dimension to a single member and removes it from scope — exactly what was needed.

✗ Rejected at formula time: "The SELECT modifier cannot filter on a Dimension that is not present in the Metric structure. Missing Dimension: CST_ActivityDest." SELECT is only valid on dimensions in the referenced metric's own structure. CST_Ctrl_Total_Overhead ($) has no CST_ActivityDest, so the modifier that would have fixed the problem is precisely the one Pigment forbids here.
3
Wrapper metric with CST_ActivityDest in its structure

Created CST_Ctrl_Total_Overhead_by_ActivityDest ($) with CST_ActivityDest in its dimension list, referencing CST_Ctrl_Total_Overhead ($), then applied [SELECT: CST_ActivityDest.Name = "Input"] in the RC01/RC06 branches.

✗ Partially improved but did not fully resolve — VAL04 still introduced ActivityDest into the outer IF's union scope, so contamination persisted from a different branch. This is the tell that the problem is not any single branch but the shared evaluation scope of the whole expression.
06 — The actual fix

The actual fix

Split the single 13-branch IF into two metrics along the natural dimensional boundary, then combine with a thin expression that introduces no dimensions of its own.

CST_Ctrl_Input_RC ($) — RC01 to RC06 only

ActivityDest enters scope only from RC04. RC04 uses [SELECT: CST_ActivityDest.Name = "Input"], which is valid because that branch's referenced metric has ActivityDest. RC01/RC06 use the wrapper metric plus [SELECT: CST_ActivityDest.Name = "Input"]. VAL04 — the other contaminating branch — is completely absent from this metric.

CST_Ctrl_Input_VAL ($) — VAL00 to VAL06 only

VAL04's ActivityDest is fully contained within this metric, where [REMOVE SUM: CST_ActivityDest] handles it correctly and consistently across the VAL branches.

CST_Ctrl_Input_Total ($) — simplified to:

'CST_Ctrl_Input_RC ($)' + 'CST_Ctrl_Input_VAL ($)'

No ActivityDest in scope at all. The combining metric is a pure sum of two already-resolved, correctly-dimensioned inputs.

07 — Core principle

The core principle

Core principle

In Pigment, any dimension referenced anywhere in an IF formula — even deep inside a nested branch — expands the evaluation scope of the entire expression. Modifiers such as SELECT and REMOVE collapse dimensions from a branch's output, but they do not prevent that dimension from entering the evaluation scope in the first place.

The only true isolation boundary is a separate metric.

Therefore: when branches of an IF operate on different dimension sets, place them in separate metrics. The combining metric should be a thin expression — a sum or coalesce — that itself introduces no new dimensions. This is the pattern that survives future edits, because adding a new mixed-dimension branch to a monolithic IF silently re-opens the contamination, whereas adding it to an isolated metric cannot reach across the boundary.

08 — Documentation gap

Documentation gap

The behaviour is partially documented, but never assembled into the causal chain that produces the failure, and never surfaced where a modeller building an IF would look.

The IF() reference — dimensional union documented, numerical consequence absent

The IF function page states that the function "automatically allocates missing Dimensions on arguments to make them match" and that "the resulting set of Dimensions is the combination of all Dimensions used in the arguments." This is the dimensional union — presented as a convenience. It never states that a branch which omits a dimension is replicated across it, that the result is then summed on assignment, or that the net effect is a value multiplied by the foreign dimension's member count.

The ×12 symptom is documented — but decoupled from IF

Troubleshoot Your Formulas notes that referencing a metric by Month when yours lacks it can make "your values … 12 times higher, depending on how many years you have set up." That is this failure's fingerprint — but it is framed around a direct metric reference, buried in a troubleshooting checklist, and never connected to IF or eager branch evaluation.

The two mechanical halves are documented — but only separately

Dimensionality Hints describes both the replication (adding a dimension uses ADD Constant, applying the value "consistently across" members) and the summation (removing a dimension uses REMOVE SUM). Nowhere are the two combined into the compounding case.

The one IF-specific hint cannot fire on a nested IF chain

Explanatory Hints surfaces a "Dimension alignment in IF expressions" hint — but only "for formulas using an IF expression, which have the 'Value if true' argument specified, but the 'Value if false or BLANK' argument not specified." Every branch in a nested IF chain specifies its False value (the next IF). The sole in-product IF dimension hint is therefore structurally incapable of triggering on the exact construct where this bites hardest.

No best-practice guidance recommends the fix

Align Block Structures for Effective Formula Operations gives only generic advice — plan your metric structure, note dimension differences, validate in the Formula Playground. No page recommends isolating mixed-dimension IF branches into separate metrics.

Summary

Every fact needed to understand this failure is documented, but scattered across four pages, framed as routine alignment, and absent from the IF() reference as a consequence. The failure passed all validation checks and was detectable only by knowing the expected output value and observing the exact 12× multiple.

09 — Scope beyond IF

Scope beyond IF

This is not unique to IF. Any construct that evaluates differently-dimensioned expressions in a shared scope exhibits the same union behaviour:

SWITCH

SWITCH uses identical eager-union evaluation across its result expressions and carries exactly the same exposure — with the same documentation silence.

Arithmetic between mismatched metrics

(MetricA + MetricB etc.) Per the "Dimension hints in arithmetic expressions" section of Explanatory Hints, operators align operands to their dimensional union in the same way.

Why IF is the most dangerous

IF is the most dangerous because the contaminating branch is the one not selected: the condition looks locally correct, the offending metric is out of view, and nothing about the winning branch hints that an unrelated branch has widened its scope. Arithmetic mismatches are usually more visible because both operands are on screen simultaneously.

10 — Questions for Pigment

Questions for Pigment

  1. Is the numerical consequence of eager evaluation — silent value inflation by a dimension's member count — documented anywhere as a consequence, beyond the scattered mechanical descriptions?
  2. Is there a plan to add a formula-time warning when an IF (or SWITCH) branch introduces a dimension not present in the target metric's structure and not referenced by all branches?
  3. Is splitting mixed-dimension IF branches into separate metrics the officially recommended pattern? If so, can it be stated on the IF() and SWITCH reference pages and in the block-structure best-practice guide?
  4. Beyond IF, SWITCH, and arithmetic operators, are there other constructs where this dimensional scope contamination can occur?

What good documentation would look like

A single note on the IF() and SWITCH reference pages:

Suggested documentation

Warning — dimensional scope across branches. IF evaluates every branch in the union of all dimensions referenced anywhere in the expression, including branches that are not selected. If one branch references a metric dimensioned by a dimension that another branch omits, the omitting branch's value is replicated across that dimension. When the result is assigned to a target metric that lacks the dimension, the replicated values are summed — producing a result equal to the correct value multiplied by the dimension's member count. Modifiers such as SELECT and REMOVE adjust a branch's output but do not remove the dimension from the shared evaluation scope. To isolate branches that operate on different dimension sets, place each set in its own metric and combine them with a thin summing metric that introduces no dimensions of its own.
